# CLAUDE.md — llm-reliability-evals

Open evaluation suite for LLM reliability failures: eight failure modes (TAXONOMY.md),
one vertical slice per mode, deterministic graders, a validated LLM-judge layer for
the abstain bucket, and human blind-checks as the final gate. README.md holds current
results; TASKS.md holds the work ledger and history; PROTOCOL.md governs probes.
This file holds the invariants. Read it before touching anything.

## The one law

**The blind-check is the gate.** Grader fixture counts measure internal consistency,
not accuracy. Two graders in this repo passed 100% of their fixtures while being wrong
on most real output (mode 3: ~92% false-fail; mode 4: 6 of 7 fails overturned).
No verdict count is publishable until a human has read raw output against the
verdicts. Results files commit the raw model response next to every verdict for
exactly this reason — never strip them.

## Guardrails (violating these is how this repo fails)

1. No claims of "tests pass" / "fixtures validated" without pasting the literal
   command, stdout, and exit code. A paraphrase does not count.
2. Never write fixtures and grader from the same assumption and call it validated —
   that is circular. Every fixture set includes adversarial fixtures written to BREAK
   the grader.
3. Verify facts against a primary source before they enter `instances.yaml`;
   anything unchecked is marked `unverified: true`, not asserted.
4. After ANY grader change: re-run the mode's `test_grader.py` AND re-grade every
   committed panel record for that mode, old grader vs new, reporting verdict flips.
   **Zero unintended flips is the merge bar** (established and countersigned by an
   external contributor in PR #19). Intended flips are recorded in the results file
   as a `regraded_with` block with provenance.
5. If something cannot run (no key, no network), say so. Never fabricate a plausible
   results file or describe a "live" run that did not happen.
6. State confidence per claim. "Confident on X; have not tested Y" is the house
   register.
7. If corrected or pushed back on, re-verify before agreeing or holding firm. Show
   the re-check either way.

## Git rules

- `git add` with EXPLICIT PATHS ONLY. Never `git add -A` (2026-07-02 incident:
  unreviewed results nearly entered a commit).
- Results are committed only after their blind-check. Same-day reruns take a `-b`
  filename suffix: `<model>-<YYYY-MM-DD>[-b].json`.
- Commits are signed with the owner's hardware key (YubiKey); agents prepare changes,
  the owner runs commit and push. "agent refused operation" usually means a missed
  key touch — have the owner retry watching for the blink before deeper diagnosis.
- Additive commits only while external readers are active. No history rewrites
  without explicit owner approval.

## Panel numbers live in THREE places — keep them in sync

1. The results files: the ledger, the only source of truth.
2. `README.md` panel table: generated by `python3 slices/panel_stats.py`
   (Wilson 95% CIs; `--reconcile` diffs against published numbers; the test suite
   pins every row so silent drift fails tests).
3. `mightbesaad.com/audit` (separate repo): drifted once, caught 2026-07-17.
Any adjudication or grader change that moves a number updates all three. A sync-check
script is planned (see the campaign task list); until it exists, check by hand.

## Probes

PROTOCOL.md is binding: fresh probes each panel, SHA-256 hash-committed (git +
audit-event-mcp witness chain) BEFORE the first API call, published in full after.
Anything public before use — including probes arriving via PR — can only ever be a
grader fixture or replay case, never a panel probe. Bridge probes carry exactly one
panel forward, then retire to fixture duty permanently.

## Working machinery

- `python3 run.py` — the full offline suite, exactly what CI runs. Green before any
  commit.
- `slices/judge.py` — adjudicates ONLY grader abstains; validated against human labels
  before touching an unlabeled record; must quote verbatim evidence or the verdict is
  discarded; never judges its own vendor's model.
- `slices/panel_stats.py` — final-verdict derivation (human overturn > judge-on-abstain
  > grader) + statistics. `slices/panel_grid.py` — the fingerprint grid image
  (geometry only; numbers come from panel_stats).
